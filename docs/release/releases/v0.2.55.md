# Changelog

## [v0.2.55] - 2026-03-31

### Fixed

- Fixed all auth redirects (`/login`, `/dashboard`, `/setup`, `/`) ignoring the configured base path. Every `res.redirect()` in `src/routes/auth.js` now goes through a `bp(res, path)` helper that prepends the base path via `res.locals.withBasePath`. Under a subdirectory reverse proxy (e.g. Nginx Proxy Manager with a `location /launcharr` block), bare redirect targets were never matched by the proxy's location block — the browser received a 404 or looped indefinitely on login, logout, and setup flows.
- Fixed the same issue in the four auth guard functions (`requireAdmin`, `requireActualAdmin`, `requireSettingsAdmin`, `requireUser`) in `src/index.js` — all now redirect to `/login` via `withBasePath`.
- Fixed `requireSettingsAdmin` path detection under a base path. The guard used `req.originalUrl` (e.g. `/launcharr/settings`) for the `startsWith('/settings')` check, which always failed when a base path was set, causing unauthenticated visitors to the settings page to receive a 403 instead of being redirected to login. The check now uses `req.url` (base-path-stripped) while still storing `req.originalUrl` in the session for the post-login redirect.

### Changed

- Pulsarr / Seerr "most-watchlisted" cards now show a coloured status pill and a Request button, matching the behaviour of the search section. The `mapMediaStatus` helper was previously inlined only in the search handler; it is now declared at module scope in `src/routes/api-media.js` so both sections share it, and the `status` field is included in the most-watchlisted API response.
- `statusColor()` in `public/pulsarr-overview.js` now handles `processing` (blue `#5b9cf6`) and `partial` (teal `#4ec9b0`) statuses. The inline ternary chain that duplicated these colours in the Seerr card builder is replaced with a call to the shared helper.

### Migration Notes

- No config or schema changes. Update and restart.
- If you use base-path hosting with Nginx Proxy Manager (or any proxy that routes by subdirectory), this release fixes the broken login/logout redirects introduced when base-path support was added. No additional proxy configuration changes are needed beyond what v0.2.54 required.
