# Changelog

## [v0.2.53] - 2026-03-29

### Fixed

- Fixed local user creation and password changes failing with `ReferenceError: hashPassword is not defined`. The `hashPassword` function was missing from the settings route context injection, so any attempt to add a new local user or change a local user's password in Settings → Users threw a server error and no changes were saved.
- Fixed custom HTTP headers not persisting after save. Headers entered in Settings → Apps → Connection were silently discarded because the form field names used `[]` suffixes (`customHeaderName[]`, `customHeaderValue[]`) which `express.urlencoded({ extended: false })` parses as literal keys with brackets — the server read the bracketed-free field names and always found an empty array. Removed the `[]` suffixes so the names match what the server expects.

### Changed

- Added debug log entries to the custom-headers save path (level `debug`, action `custom-headers-save`) so the raw parsed names and final header count are visible in the activity log if the fix needs verification.

### Migration Notes

- No schema migration required.
- Any custom HTTP headers that were previously entered but not saved correctly will need to be re-entered after upgrading.

### Verification Checklist

- [ ] Add a new local user in Settings → Users — should succeed without error.
- [ ] Change a local user's password — should succeed without error.
- [ ] Add one or more custom HTTP headers to an app in Settings → Apps → Connection and save — headers should persist on page reload and appear in `config.json`.
- [ ] Debug log entries for `custom-headers-save` appear in the activity log after a headers save.
